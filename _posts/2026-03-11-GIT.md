# git pull 和 rebase

git pull和本地处在什么分支强相关：

- git pull会根据当前的分支，直接拉取upstream
- git pull origin main把远程main分支的内容合并到当前分支
- git pull origin feat把远程feat分支的内容合并到当前分支

git pull默认是merge合并，最好用rebase，在git pull加上参数 **--rebase**



git fetch origin只会更新远程的最新提交（所有分支）下来，不会改动本地

## 复制一个branch？

假设remote是main，想接着在feat开发：

```bash
$ git pull origin main# main->main
$ git switch -c feat# 等价于git checkout -b feat
$ git rebase main
```



## 只操作一个branch？

`git pull --rebase origin dev/vl` 的意思是：

1. **从远端 `origin` 拉取（fetch）** 分支 `dev/vl` 的最新提交
2. 然后把你**当前所在分支**上“你自己的本地提交”**rebase 到** `origin/dev/vl` 的最新提交之上。

等价理解为两步（概念上）：

```
git fetch origin dev/vl
git rebase origin/dev/vl
```

------

## 和你当前所在分支有没有关系？

**有，而且关系很大。**

- 你执行这条命令时，Git 会对你**当前 checkout 的分支**做 rebase。
- `origin dev/vl` 只是告诉 Git：把远端哪个分支当作“基底”（upstream）。

所以它不是“把 dev/vl 拉到 dev/vl”，而是：

> “把 **当前分支** 更新到基于 `origin/dev/vl` 的最新状态，并尽量保持线性历史”。

------

## 举个例子

你现在在 `feat` 分支：

- `origin/dev/vl` 有新提交 A-B-C
- 你本地 `feat` 有自己的提交 x-y

跑：

```
git pull --rebase origin dev/vl
```

结果会变成：

- 先把远端 A-B-C 拉下来
- 再把你的 x-y “搬到” C 后面：A-B-C-x'-y'

# create repo

## 1) 本地初始化 git（如果还没初始化）

```bash
cd /path/to/your-project
git init
git add .
git commit -m "Initial commit"
```

> 如果你已经有 git 提交历史了，就从下一步开始。

## 2) 在 GitHub 上创建一个空仓库

去 GitHub → 右上角 **+** → **New repository**

- 填仓库名
- 建议先选 **Private**（可后面改）
- **不要勾选** README / .gitignore / license（避免和本地历史冲突）
  创建完成后，GitHub 会给你一个仓库地址（HTTPS 或 SSH）。

## 3) 绑定远端并推送

### 方式 A：HTTPS

```bash
git remote add origin https://github.com/<yourname>/<repo>.git
git branch -M main#统一默认分支是main
git push -u origin main
```

### 方式 B：SSH

1. 先确保你本机有 SSH key：

```bash
ls ~/.ssh/id_ed25519.pub
```

没有的话生成：

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

1. 把 `~/.ssh/id_ed25519.pub` 内容粘到 GitHub：Settings → SSH and GPG keys
2. 推送：

```bash
git remote add origin git@github.com:<yourname>/<repo>.git
git branch -M main
git push -u origin main
```

## 常见坑

- **push 被拒绝 / 要登录**：HTTPS 现在用 **Personal Access Token** 作为密码（不是账号密码）。
- **仓库里已有 README 导致冲突**：删掉远端内容重建空仓库，或 `git pull --rebase` 解决冲突后再 push。
- **默认分支不是 main**：用 `git branch -M main` 统一。





# 创建新分支，并推送新分支？

## 1) 确认你当前在哪个分支、改了什么

```bash
git status
git branch --show-current
```

## 2) 基于当前分支创建并切到新分支

（分支名自己取，比如 `feat/eau-opt`）

```bash
git switch -c feat/eau-opt
# 老版本 git 用：
# git checkout -b feat/eau-opt
```

## 3) 提交你的修改

```bash
git add -A
pre-commit run --all-files
git commit -m "Describe your change"
```

## 4) 推送新分支到远端（remote）

```bash
git push -u origin feat/eau-opt
```

`-u` 会建立 upstream，以后你在这个分支上直接 `git push` / `git pull` 就行。

------

### 常见情况/坑

**A) 你还没配置 remote**

```bash
git remote -v
```

没有 `origin` 的话：

```bash
git remote add origin <remote-url>
```

**B) 远端默认分支更新了，你想基于最新再推**
先拉最新：

```bash
git fetch origin
git rebase origin/main   # 如果主分支叫 main
# 或者 origin/master
```

有冲突就解决后：

```bash
git rebase --continue
```

然后再 push。

**C) 你还没权限往原仓库 push（常见于公司仓/开源仓）**
会报 `permission denied` 或 `protected branch`。这时通常要：

- 走 **fork → 推送到你自己的 fork**，再提 PR；或
- 让管理员给你权限 / 让你推到指定分支。

## git commit -c HEAD

git commit -c HEAD 是一个非常有用的命令，它的核心作用是：复用当前分支最后一次提交（HEAD）的信息作为模板，并允许你进行修改后创建一个新的提交。

这里的 -c（小写）代表 --reedit-message。

1. **基本用法步骤**

假设你刚刚完成了一个提交，现在又做了一些改动，想针对这些改动发一个“类似”的提交。

暂存修改：

code

Bash

git add .

执行命令：

code

Bash

git commit -c HEAD

编辑并保存：

Git 会自动打开你配置的文本编辑器（如 vim 或 nano），里面已经填好了上一个提交的消息（Commit Message）。你可以直接修改、添加内容，保存后退出，一个新的提交就产生了。

2. **它与 git commit --amend 的区别**

这是最容易混淆的地方：

git commit --amend：

作：修改上一次提交。它不会产生新提交，而是替换掉旧的提交。

场景：发现刚才的代码少改了一行，或者提交信息写错了。

git commit -c HEAD：

作用：创建一个全新的提交。它只是把上一次的提交信息“借”过来当草稿。

场景：你正在做一系列相关的任务（例如“修复 Bug A - 第一部分”、“修复 Bug A - 第二部分”），你希望第二次提交的消息格式和第一次保持高度一致，免得重新打字。

3. -c (小写) vs -C (大写)

这两个参数的区别在于是否打开编辑器：

-c HEAD (小写)：复用信息并打开编辑器供你修改（推荐）。

-C HEAD (大写)：直接复用信息并立刻完成提交，不会让你修改文字。

注意：如果你不加修改地复用，通常会导致两次提交的信息一模一样，这在日志里可能会让人困惑。

4. 常见应用场景

系列提交：当你需要连续提交好几个功能模块，且它们的描述前缀相同（例如都包含 [Feature-123] ...）时。

修正后的新提交：由于某些原因你不能使用 amend（例如旧提交已经推送到远程共享分支），你只能发一个新提交，但又想参考之前的描述。

总结

如果你想**“复制上个提交的消息并改一改，然后发个新提交”**，就用 git commit -c HEAD。

# git commit --amend相关问题

假设你最开始：

```bash
A
```

你 push 之后，远端也是：

```bash
A
```

然后你本地执行：

```bash
git commit --amend
```

本地就不再是原来的 `A` 了，而变成一个**新的提交**，比如：

```bash
B
```

这时状态就是：

- 远端：`A`
- 本地：`B`

注意，`B` **不是** 在 `A` 后面追加的提交，
而是“用一个新提交替换掉了原来的 A”。

所以这不是普通的“往前快进”关系。

------

## 为什么普通 `git push` 会让我先 pull

普通 `git push` 默认只允许 **fast-forward**。

也就是远端历史必须是你本地历史的前缀。
例如这种可以直接 push：

```bash
远端: A
本地: A -> C
```

因为只是往后加了一个 `C`。

但 amend 后变成：

```bash
远端: A
本地: B
```

这里远端的 `A` 并不是本地 `B` 的祖先。
Git 会认为：

> 你要把远端当前的提交历史覆盖掉

这属于 **non-fast-forward push**，所以它会拒绝，并提示你先 pull。

这个“先 pull”其实只是 Git 的通用提示，不一定真的适合这个场景。
因为你这里不是想合并远端更新，而是想**改写远端历史**。

------

## 为什么 `-f` 或 `--force-with-lease` 可以

因为它们的意思就是：

> 我知道这不是 fast-forward，我就是要覆盖远端。

所以 amend/rebase 之后，通常都需要：

```bash
git push --force-with-lease
```

而不是普通：

```bash
git push
```

------

## 那为什么不是先 pull 再 push

因为 pull 的作用是把远端历史合并到本地。

但你这个场景里，远端只有旧的 `A`，本地是新的 `B`。
如果你先 pull，Git 会尝试把两边协调起来，反而会让历史变得奇怪，通常不是你想要的。

你真正想做的是：

- 远端原来的 `A` 不要了
- 用本地新的 `B` 替换它

这就应该 force push。

------

## `-f` 和 `--force-with-lease` 的区别

### `-f` / `--force`

直接强推，不太管远端现在是不是被别人更新了。

### `--force-with-lease`

更安全。它会先检查：

> 远端还是不是我以为的那个状态？

如果远端这期间被别人改了，它会拒绝，避免你把别人的提交冲掉。

所以平时更推荐：

```bash
git push --force-with-lease
```

------

普通 `push` 只接受：

> 在原历史后面继续长

而 `amend` 做的是：

> 把原来的提交换成一个新的

所以普通 `push` 会拒绝；
`force push` 才允许你把远端旧提交替换掉。

# 怎么退回以前的提交？

```bash
git revert HEAD
```

作用：

- 撤销最近一个 commit 的内容
- 但保留历史
- 会新生成一个 `revert commit`

如果这个提交已经推到远端：

```bash
git push
```

后来如果决定把“原 commit + revert commit”都从历史里删掉，就进入交互式 rebase：

```bash
git rebase -i HEAD~3
```

在打开的编辑器里，把这两条从：

```text
pick <sha1> feat...
pick <sha2> revert...
```

改成：

```text
drop <sha1> feat...
drop <sha2> revert...
```

作用：

- 重写本地历史
- 让这两个 commit 彻底消失

>  如果 rebase 过程中状态混乱，先退出：
>
> git rebase --abort

作用：

- 放弃当前 rebase
- 回到 rebase 之前的状态
- 再重新执行 `git rebase -i ...`

rebase 成功后检查状态：

```bash
git status
git log --oneline -5
```

作用：

- 确认本地工作区干净
- 确认那两个 commit 已经从本地历史消失

如果这时看到：

```text
Your branch is behind origin/... by 2 commits
```

并且确认这 2 个 commit 正是刚刚 drop 掉的那两个，那么不要 `git pull`，否则会把它们又拉回来。

应该执行：

```bash
git push --force-with-lease
```

作用：

- 用本地重写后的历史覆盖远端
- 让远端也同步删掉那两个 commit

完整链路就是：

```bash
git revert HEAD
git push

git rebase -i HEAD~3
# 把原 commit 和 revert commit 改成 drop

git status
git log --oneline -5

git push --force-with-lease
```
